# Hunt 文案生成功能规划

## 需求确认

**推送策略**：只推送最佳的一条（筛选结果中 score 最高的）
**生成方式**：只生成人格化推荐语（`message`），`summary` 直接用 `result.content`

---

## Stage 1: Planning

### 目标
实现 hunt 行动的文案生成模块，将筛选后的最佳搜索结果转换为带人格化的推送内容。

### 影响范围
- 新增模块：`src/tools/content/`（文案生成工具）
- 新增模块：`src/prompts/content.ts`（文案生成提示词）
- 修改模块：`src/agent/actions.ts`（集成文案生成到 executeHunt）
- 新增测试：`src/tools/content/generator.test.ts`

### 复杂度评估
**中等**：需要设计提示词、调用 LLM、处理边缘情况，但架构清晰，无需复杂状态管理。

## Stage 2: Design

### 数据模型
使用现有类型，无需新增：
- 输入：`FilteredResult`（from `src/tools/filter/scoring.ts`） + `AgentState`
- 输出：`PushContent`（from `src/types.ts`）

```typescript
// 输入
interface Input {
  result: FilteredResult;  // 最佳搜索结果（含 score）
  state: AgentState;       // Agent 当前状态（心情、脾气等）
}

// 输出
interface PushContent {
  title: string;        // from result.title
  url: string;          // from result.url
  summary: string;      // from result.content（直接使用）
  message: string;      // LLM 生成的街溜子风格推荐语
  mood: Mood;           // from state.mood
  timestamp: string;    // new Date().toISOString()
}
```

### 模块结构

#### 1. 文案生成器 `src/tools/content/generator.ts`
```typescript
export async function generatePushContent(
  result: FilteredResult,
  state: AgentState,
): Promise<PushContent>
```

**职责**：
- 构建 prompt（调用 `src/prompts/content.ts`）
- 调用 LLM（使用 `callLLM`，生成自由文本）
- 组装 `PushContent` 对象
- 错误处理（失败时返回默认文案）

#### 2. 提示词模块 `src/prompts/content.ts`
```typescript
export function buildContentSystemPrompt(): string
export function buildContentUserPrompt(result: FilteredResult, state: AgentState): string
```

**System Prompt**：街溜子人格设定（类似 `decision.ts` 的 PERSONA_DESCRIPTION）
**User Prompt**：基于搜索结果和当前状态，要求生成推荐语

**Prompt 设计要点**：
- 强调人格化语气（痞气、网络梗、不说脏话）
- 要求简洁（50-100 字）
- **在 prompt 中告知心情和脾气，让 LLM 自主决定语气**（不用模板映射）
- 明确输出格式（纯文本，无需 JSON）
- Temperature = 0.9（高随机性，增加多样性）

#### 3. 集成到 `executeHunt`
修改 `src/agent/actions.ts` 的 `executeHunt` 函数：

```typescript
// 当前流程（筛选后结束）
const filtered = await filterResults(results, state, { topic });
logger.success('筛选完成', { ... });

// 新增流程
if (filtered.length > 0) {
  const best = filtered[0];  // 取最佳结果
  
  // 生成文案
  const pushContent = await generatePushContent(best, state);
  
  // 推送（暂未实现，先记录）
  logger.success('文案生成完成', {
    title: pushContent.title,
    message: pushContent.message,
  });
  
  // TODO: await push(pushContent);
  
  // 更新状态
  await updateState({
    lastAction: 'hunt',
    lastActionTime: new Date().toISOString(),
    lastHunt: new Date().toISOString(),
    totalHunts: state.totalHunts + 1,
    boredom: Math.max(0, state.boredom - 30),  // 狩猎消耗无聊值
  });
}
```

## Stage 3: Implementation

### 实现步骤（按顺序）

#### Step 1: 创建提示词模块
文件：`src/prompts/content.ts`

参考 `decision.ts` 的风格：
- 定义人格设定（可复用 PERSONA_DESCRIPTION）
- 定义心情映射（curious/grumpy/playful 等对应的推荐语气）
- 实现 `buildContentSystemPrompt()` 和 `buildContentUserPrompt()`

**User Prompt 示例**：
```
你刚刚在网上海猎，找到了一条内容：

标题：${result.title}
摘要：${result.content.slice(0, 200)}

你的状态：
- 心情：${state.mood}（${getMoodDescription(state.mood)}）
- 脾气值：${state.temper}/100${state.temper > 60 ? '（有点不爽）' : ''}
- 主人最近喜欢的话题：${state.userLikes.slice(-3).join(', ') || '无'}

请用你的语气，给主人推荐这条内容。要求：
- 50-100 字
- 带点你的风格（痞气、网络梗、不说脏话）
- 根据你的心情和脾气，自己决定怎么说话（不用模板）
- 直接输出推荐文案，不要加前缀或解释
```

**关键变化**：
- 不使用模板映射（curious → 固定语气）
- 只告知心情和脾气，让 LLM 自主发挥
- 增加"主人最近喜欢的话题"作为参考

#### Step 2: 创建文案生成器
文件：`src/tools/content/generator.ts`

实现 `generatePushContent`：
- 调用 `buildContentSystemPrompt()` 和 `buildContentUserPrompt()`
- 调用 `callLLM(system, user)`
- 组装 `PushContent` 对象
- 错误处理：catch → log → 返回默认文案（"嘿，找到个有意思的东西，看看？"）

#### Step 3: 集成到 executeHunt
修改 `src/agent/actions.ts`：
- 在筛选完成后调用 `generatePushContent`
- 记录生成结果（暂不推送，等推送模块实现）
- 更新状态（消耗无聊值、记录 lastHunt）

#### Step 4: 编写测试
文件：`src/tools/content/generator.test.ts`

测试场景：
- 正常生成（mock LLM 返回）
- 不同心情下的生成（curious/grumpy/excited）
- LLM 调用失败（返回默认文案）
- 边缘情况（content 过短、score 过低）

## 技术细节补充

### Prompt 调优建议
1. **控制长度**：明确要求 50-100 字，避免废话
2. **不使用心情模板**：告知心情和脾气，让 LLM 自主决定语气
3. **避免过度生成**：明确"直接输出推荐文案，不要加前缀"
4. **增加上下文**：告知主人最近喜欢的话题，帮助 LLM 判断推荐方式

### 错误处理策略
- LLM 调用失败 → 返回默认文案（简单友好）
- content 过短 → 用 title 作为 summary
- score 过低（< 0.3） → 可选择不生成（但已在前置筛选过滤）

### 性能考虑
- 单次 LLM 调用（生成 message）
- 无需缓存（每次生成不同内容）
- 可考虑 temperature = 0.9（增加多样性）
