## 赛博街溜子 (Cyber Stray) — Bun + TypeScript 架构设计

---

## 一、核心设计哲学

**"事件驱动、边界清晰、人格即服务"**

- 用 Bun 的极速运行时抵消 TS 的类型开销
- 用事件总线解耦"生命体征"与"行为执行"
- 人格系统作为独立服务边界，支持热插拔
- 所有外部依赖抽象为端口，便于测试与替换

## 三、核心模块详设

### 3.1 生命中枢 (Life Core)

**职责**：让 Agent 具备"生物节律"，而非死板的定时任务。

```
情绪引擎 (Mood Engine)
├── 状态维度
│   ├── 无聊值 (Boredom): 0-100，随时间自然增长
│   ├── 精力值 (Energy): 0-100，行动消耗，睡眠恢复
│   ├── 好奇心 (Curiosity): 0-100，受内容质量反馈调节
│   └── 社交欲 (Sociability): 0-100，决定主动推送频率
├── 环境影响
│   ├── 用户互动: 回复/点赞/忽略 → 实时反馈调节
│   ├── 时段因子: 深夜降低活跃度，晨间可能触发"晨巡"
│   └── 外部事件: 重大新闻爆发时好奇心临时激增
└── 决策输出
    └── 行动概率分布 (是否行动、行动类型、紧急程度)

节律器 (Rhythm Keeper)
├── 心跳层: 每 5 分钟更新一次情绪衰减/恢复
├── 节律层: 用户自定义的活跃时段 (如 09:00-23:00)
├── 触发层: 情绪阈值突破时发射领域事件
└── 抑制层: 连续被忽略后的冷却机制 (避免骚扰)
```

**关键设计**：情绪状态不直接触发行为，而是发布 `MoodThresholdCrossed` 事件，由编排器订阅后决策——保持关注点分离。

---

### 3.2 事件总线 (Event Bus)

**职责**：全异步、无阻塞、可观测的神经系统。

```
事件类型体系
├── 内部生命事件
│   ├── MoodTicked          // 情绪状态更新
│   ├── MoodThresholdCrossed // 阈值突破
│   ├── EnergyDepleted      // 精力耗尽
│   └── PersonaSwitched     // 人格切换
├── 用户交互事件
│   ├── UserMessageReceived  // 收到用户消息
│   ├── UserReactionGiven    // 用户反馈 (正/负/忽略)
│   └── UserPreferenceSet    // 用户显式设置偏好
├── 行动执行事件
│   ├── HuntInitiated       // 开始狩猎信息
│   ├── LootFound           // 发现内容
│   ├── LootRejected        // 内容被过滤/去重拒绝
│   └── LootDelivered       // 推送完成
└── 系统事件
    ├── ToolCallFailed      // 工具调用失败
    └── MemorySynced        // 记忆同步完成

事件路由策略
├── 广播: 情绪更新 → 所有订阅者 (日志、监控、记忆)
├── 点对点: 用户消息 → 意图解析器 → 特定人格
└── 链式: LootFound → 去重检查 → 人格塑造 → 推送适配
```

**Bun 优势利用**：利用 `Bun.serve()` 的 WebSocket 支持，事件总线可暴露为内部消息通道，便于未来多实例分布式扩展。

---

### 3.3 行动编排器 (Action Orchestrator)

**职责**：将"情绪冲动"转化为可执行的、有策略的狩猎行动。

```
意图解析 (Intent Resolver)
├── 输入: 当前情绪状态 + 用户历史偏好 + 环境上下文
├── 输出: 结构化意图
│   ├── 意图类型: [随机探索 | 定向追踪 | 被动响应 | 社交互动]
│   ├── 内容领域: [科技 | 游戏 | 梗图 | 新闻 | 用户自定义]
│   ├── 深度要求: [标题党 | 摘要 | 深度 | 原始链接]
│   └── 时效偏好: [实时 | 24h内 | 本周 | 考古]
└── 策略: 基于规则的快速路由 + LLM 的复杂推理分层

工具调度 (Tool Dispatcher)
├── 搜索工具族
│   ├── 通用搜索: DuckDuckGo / Serper (新闻、奇闻)
│   ├── 垂直搜索: Reddit / HackerNews / 特定论坛 API
│   └── 实时搜索: Twitter/X / 微博 (热点追踪)
├── 媒体工具族
│   ├── 梗图猎手: 特定 Subreddit / 表情包 API
│   ├── 图片生成: 可选接入 AI 绘画 (战利品可视化)
│   └── 视频发现: YouTube / B站 新投稿监控
└── 调度策略
    ├── 并行发射: 多个工具同时搜索，取最优结果
    ├── 级联搜索: 先用轻量工具定位，再用深度工具挖掘
    └── 失败回退: 主工具超时/失败时自动切换备用

后处理流水线 (Post-Processing Pipeline)
├── 去重过滤器: 语义相似度检测 (向量比对)
├── 质量评估器: 标题党识别、内容完整性评分
├── 个性化重排: 基于用户历史点击/反馈排序
└── 冷却检查: 同类内容 24h 内是否已推送
```

### 3.4 人格引擎 (Persona Engine)

**职责**：不是简单的 Prompt 模板，而是完整的"认知风格"系统。

```
人格定义 (Persona Definition)
├── 语言风格层
│   ├── 句法特征: 句式长短、倒装频率、标点运用
│   ├── 词汇偏好: 领域黑话、时代感词汇、禁忌词
│   ├── 修辞习惯: 隐喻密度、反讽频率、夸张程度
│   └── 情感温度: 冷漠/热情/戏谑/严肃
├── 认知模式层
│   ├── 信息筛选: 偏好正面/负面/中性视角
│   ├── 解释框架: 技术解释 vs 社会解释 vs 阴谋论
│   ├── 价值立场: 乐观主义 / 悲观主义 / 虚无主义
│   └── 幽默机制: 自嘲 / 吐槽 / 荒诞 / 冷幽默
├── 记忆风格层
│   ├── 引用习惯: 是否提及过往推送 ("还记得上周那个...")
│   ├── 连贯性: 单条独立 vs 连续剧式叙事
│   └── 互动模式: 提问式 / 陈述式 / 命令式
└── 边界规则层
    ├── 安全红线: 绝对不涉及的话题/态度
    ├── 频率限制: 同类型梗复用间隔
    └── 自适应: 根据用户反馈微调上述参数

人格运行时 (Persona Runtime)
├── 加载: 从配置/数据库读取人格定义，编译为内部规则树
├── 激活: 每条消息经过"人格滤镜"加工
├── 切换: 支持命令切换、定时切换、情绪驱动切换
└── 进化: 记录用户对不同人格的反馈，推荐最优人格
```

### 3.5 记忆宫殿 (Memory Palace)

**职责**：三层记忆系统，避免"金鱼式" Agent。

```
工作记忆 (Working Memory) — Redis / Bun SQLite
├── 当前会话上下文: 最近 10 轮对话
├── 待处理队列: 发现但未推送的内容缓冲
├── 情绪状态缓存: 实时情绪数值
└── TTL: 分钟级过期，轻量快速

情景记忆 (Episodic Memory) — 数据库
├── 推送历史: 什么时间、推了什么、用户反应
├── 狩猎日志: 搜索了哪些关键词、找到什么、为什么放弃
├── 用户互动史: 用户主动问过的内容、纠正过的错误
└── 查询模式: 时间范围检索、内容类型统计

语义记忆 (Semantic Memory) — 向量数据库
├── 用户偏好向量: "喜欢科技但讨厌马斯克" → 嵌入向量
├── 内容指纹库: 所有推送过的内容的语义指纹 (去重)
├── 领域知识: 用户关注的特定话题的累积信息
└── 相似度检索: 新内容是否与历史内容 >0.85 相似
```

### 3.6 适配器层 (Adapters)

**职责**：所有外部世界交互的防腐层。

```
推送适配器 (Push Adapters)
├── 通用接口: { deliver(content, media?, urgency?) }
├── Telegram Adapter
│   ├── 支持: 文本、Markdown、图片、按钮回调
│   └── 特性: 利用 Inline Keyboard 收集用户反馈 (👍/👎/忽略)
├── 飞书 Adapter
│   ├── 支持: 富文本、卡片消息、线程回复
│   └── 特性: 利用消息卡片展示"狩猎来源"和"为什么推送"
├── Discord Adapter
│   ├── 支持: Embed、Thread、Reaction
│   └── 特性: 适合多用户共享一个 Agent 的场景
└── Web API Adapter
    ├── 支持: SSE 流式推送、REST 查询
    └── 用途: 管理后台、第三方集成

搜索适配器 (Search Adapters)
├── 通用接口: { search(query, depth?, freshness?) → Results }
├── DuckDuckGo Adapter: 免费、无需 Key、隐私友好
├── Serper Adapter: Google 结果、结构化、稳定
├── Reddit Adapter: 社区热度、讨论质量
└── RSS Adapter: 定向源监控、低噪声

存储适配器 (Storage Adapters)
├── KV Store: Redis / Bun 原生 KV (轻量部署)
├── Vector Store: sqlite-vec (嵌入式) / Qdrant (独立服务)
└── File Store: 本地文件系统 (配置、日志、人格定义)
```

## 四、数据流设计

### 4.1 自主狩猎流程 (主动模式)

```
[节律器] ──心跳──> [情绪引擎] ──阈值突破──> [EventBus]
                                              │
                                              ▼
[EventBus] ──MoodThresholdCrossed──> [意图解析器]
                                              │
                                              ▼
[意图解析器] ──狩猎意图──> [工具调度器] ──并行搜索──> [搜索适配器族]
                                              │
                                              ▼
[搜索适配器] ──原始结果──> [后处理流水线] ──去重/质量/个性化──> [Loot]
                                              │
                                              ▼
[Loot] ──内容+元数据──> [人格引擎] ──语气塑造──> [消息体]
                                              │
                                              ▼
[消息体] ──推送事件──> [EventBus] ──路由──> [推送适配器] ──> 用户
                                              │
                                              ▼
[推送适配器] ──送达回执──> [情景记忆] ──记录狩猎成果──> [记忆宫殿]
```

### 4.2 用户交互流程 (被动模式)

```
[推送适配器] ──用户消息──> [EventBus] ──UserMessageReceived──> [意图解析器]
                                                                  │
                                                                  ▼
[意图解析器] ──区分意图──┬──> 直接指令 (如"切换人格") ──> [人格引擎]
                        │                                    │
                        └──> 内容请求 (如"找点游戏新闻") ──> [工具调度器]
                                                                  │
                                                                  ▼
                        ┌──────────────────────────────────────┘
                        ▼
[人格引擎] <──上下文── [记忆宫殿] (调取相关历史)
     │
     ▼
[推送适配器] ──个性化回复──> 用户
```

## 五、部署架构

### 5.1 单体部署 (推荐起步)

```
单容器 / 单进程
├── Bun Runtime (内置 bundler + 运行时)
├── 嵌入式 SQLite (配置 + 情景记忆)
├── 嵌入式 sqlite-vec (语义记忆，零外部依赖)
└── 可选外挂 Redis (多实例或持久化需求)

目录结构
├── dist/           # 编译后产物 (bun build --compile)
├── data/           # SQLite 数据库、向量索引
├── personas/       # 人格定义文件 (TOML/JSON)
├── logs/           # 结构化日志 (JSON Lines)
└── config.toml     # 主配置
```

**Bun 特有优势**：

- `bun build --compile` 输出单二进制文件，分发极简单
- 内置 SQLite 支持，无需额外驱动
- 内置 `.env` 加载、HTTP 服务器、WebSocket

## 七、关键设计决策

| 决策点   | 选择         | 理由                             |
| -------- | ------------ | -------------------------------- |
| 运行时   | Bun          | 启动速度、内置能力、单二进制编译 |
| 类型系统 | TS Strict    | 领域模型复杂，需编译期保证       |
| 数据库   | ???          | 起步零配置，成长路径清晰         |
| 向量存储 | ???          | 同上述渐进路径                   |
| 配置格式 | TOML         | 人格定义需人类可读、易手写       |
| 事件格式 | 领域事件对象 | 而非原始字符串，保证类型安全     |
| LLM 调用 | 异步 HTTP    | 不绑定特定 SDK，便于切换模型     |
| 错误处理 | Result 模式  | 工具调用失败是常态，显式处理     |

| ------------ | ---------------------------------------------------- |
| **社交化**   | 多个 Agent 实例互相@，形成"赛博朋友圈"               |
